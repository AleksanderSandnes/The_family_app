# The Family App — iOS

Native SwiftUI port of the Android app with 1:1 feature and design parity. Talks to the
**same Supabase project** (Auth, Postgrest, Realtime, Storage) as `android/`.

## Mac setup

```bash
brew install xcodegen
cd ios
cp Config/Secrets.example.xcconfig Config/Secrets.xcconfig   # fill in real values
xcodegen generate
open FamilyApp.xcodeproj
```

- `Secrets.xcconfig` needs the same `SUPABASE_URL` / `SUPABASE_ANON_KEY` as
  `android/local.properties`. Mind the `$()` escape in the URL (xcconfig comment rule).
- `FamilyApp.xcodeproj` is **generated** — never commit it. Re-run `xcodegen generate`
  after editing `project.yml`.
- Set your team under Signing & Capabilities locally (not committed).

### Push notifications (physical device)

1. Firebase console: add an **iOS app** with bundle id `com.sandnes.familyapp`, upload the
   APNs auth key (.p8), download `GoogleService-Info.plist` →
   `ios/FamilyApp/Resources/` (gitignored). Firebase only activates when the plist exists.
2. Redeploy the platform-aware edge functions:
   `supabase functions deploy push-on-message daily-reminders`.
3. The `aps-environment` entitlement is generated by XcodeGen; Xcode manages the
   dev/production value at signing time. Push requires a physical device.

Command-line build & tests:

```bash
xcodebuild -project FamilyApp.xcodeproj -scheme FamilyApp \
  -destination 'platform=iOS Simulator,name=iPhone 16' build
xcodebuild -project FamilyApp.xcodeproj -scheme FamilyApp \
  -destination 'platform=iOS Simulator,name=iPhone 16' test
```

## ⚠️ The dual-ID rule (copy of the Android warning — it bites silently)

Every user has **two different UUIDs**:

- **`public.users.id`** — the app-internal user id. Stored by `SessionStore`
  (`current_user_id_v2`). Used in ALL foreign keys (`shopping_lists.owner_user_id`,
  `calendar_events.user_id`, …).
- **`auth_id` / `auth.uid()`** — the Supabase Auth UUID, from
  `SupabaseClientProvider.client.auth.currentSession?.user.id`. Used by RLS policies
  and **Storage paths** (`avatars/{auth_uid}/…` — the bucket RLS checks
  `(storage.foldername(name))[1] = auth.uid()`).

Mixing them up causes silent RLS denials or rows invisible to the family. All storage
uploads must go through `StorageService`, which resolves the auth UUID itself.

## Architecture map (Android → iOS)

| Android | iOS |
|---|---|
| `SupabaseManager` | `Core/SupabaseClientProvider` (PKCE, `familyapp://auth`, 25s heartbeat) |
| `SessionManager` (DataStore) | `Core/SessionStore` (UserDefaults, identical keys) |
| `FamilyRepository.get(context)` | `FamilyRepository.shared` |
| `Entities.kt` | `Core/Models/Entities.swift` (same snake_case columns) |
| `postgresChangeFlow` + `"table-$familyId"` channel + full reload | `Core/RealtimeObserver` (same channel names, same reload-on-any-event) |
| Storage path helpers in ViewModels | `Core/StorageService` (auth-uid enforced) |
| Coil | Nuke (`ImagePipelineConfig`, `LazyImage`) |
| `ui/theme/` (Color/Type/Shape/Spacing) | `DesignSystem/` (identical hex/size/radius tokens) |
| `ui/components/` | `DesignSystem/Components/` (same component names) |
| NavHost + bottom nav | `TabView` + per-tab `NavigationStack` (`Features/Navigation`) |
| WorkManager reminders | none — server-side `daily-reminders` (pg_cron) is client-agnostic |

Feature code lives in `FamilyApp/Features/<Domain>/` mirroring `ui/<domain>/` on Android.
Unit tests live in `FamilyAppTests/` and run with the scheme's test action.

## Realtime / optimistic-UI conventions (same as Android)

1. Channel per table+family: `client.channel("tablename-\(familyId)")`.
2. Any event → **full reload**, never a partial merge.
3. Teardown on disappear (`RealtimeObserver.stop()`).
4. Writes: update local state immediately (temp `UUID()`), fire the call, reload from
   server. Deletes remove before the network call. No rollback logic.
